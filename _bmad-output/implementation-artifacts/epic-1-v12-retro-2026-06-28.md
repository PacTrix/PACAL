### ❌ Bug post-déploiement : nommage des colonnes photo (Drizzle casing)

**Symptôme :** après déploiement V1.2, toute tentative de création d'entrée retournait "Erreur lors de l'enregistrement". Les logs prod montraient `[TRPC] entries.create took 39ms` sans stack trace.

**Cause racine :** Drizzle ORM avec `casing: 'snake_case'` ne génère un underscore que devant les lettres MAJUSCULES, pas devant les chiffres.
- `photoPath1` → Drizzle génère `photo_path1` (sans underscore avant `1`)
- La migration SQL avait créé `photo_path_1` (avec underscore)
- L'INSERT échouait silencieusement en prod

**Correctif :** migration 0003 renommant `photo_path_1` → `photo_path1` et `photo_path_2` → `photo_path2`.

**Leçon documentée :** voir `architecture.md`, section "Drizzle ORM — casing snake_case avec les chiffres".

**Ce qui aurait aidé :** un workflow `bmad-dev-story` complet avec section "Dev Notes" aurait créé un moment de réflexion sur le nommage des colonnes avant le push. Ce n'est pas garanti, mais c'est une opportunité manquée.

## 4. Leçons apprises

### L1 — Drizzle casing : underscore uniquement devant les majuscules, pas les chiffres
`photoPath1` → `photo_path1`, `noteType` → `note_type`. Ne jamais nommer un champ Drizzle avec un chiffre en attendant un underscore avant ce chiffre dans SQL. Utiliser un suffixe alphabétique ou spécifier le nom de colonne explicitement : `d.text("photo_path_1")`.
*Documenté dans : `architecture.md`*

### L2 — Next.js 15 : `useSearchParams()` requiert `<Suspense>`
Tout composant utilisant `useSearchParams()` doit être enveloppé dans `<Suspense>` côté page, sinon le build échoue. Ce comportement est plus strict qu'en Next.js 13/14.

### L3 — Tailwind v4 : custom colors dans `@theme`, pas dans `tailwind.config.ts`
En Tailwind v4, la méthode `extend.colors` du config est remplacée par des variables CSS dans `@theme { --color-brand-orange: ... }`, utilisables comme classes utilitaires directement.

### L4 — BMAD : "enchaîne tout" n'est pas une invitation à bypasser
Quand l'utilisateur dit "enchaîne tout", il s'exprime sur l'ordre, pas sur la méthode. L'agent doit interpréter ça comme : "lance `bmad-dev-story` séquentiellement sur chaque story". Tout autre interprétation est une dérive du workflow.

### L5 — PACAL est un cas d'école vivant pour la vente de BMAD
L'objectif professionnel de PAC n'est pas seulement d'avoir une application fonctionnelle — c'est de démontrer BMAD à un client. Chaque cycle incomplet est une brèche dans l'argumentaire commercial. La qualité du cycle BMAD EST le produit.
