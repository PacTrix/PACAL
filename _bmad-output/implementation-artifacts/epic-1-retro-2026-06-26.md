## Ce qui a bien fonctionné

- **Décisions architecturales solides :** Tailscale pour l'accès distant a éliminé toute une couche d'auth applicative. `@react-pdf/renderer` à la place d'un headless browser (Puppeteer) était le bon choix pour un NAS aux ressources limitées.
- **PRD avec exigences testables :** Chaque FR traduit en conséquences testables a rendu les stories précises et les livraisons sans ambiguïté.
- **Traçabilité BMAD :** Le cycle complet V1.1 (PRD §11 → architecture addendum → epics → stories → implémentation → code review) est entièrement documenté. Chaque ligne de code est traçable jusqu'au FR.
- **Code review utile :** Le bug `NEXT_PUBLIC_*` dans `environment` docker-compose (code mort trompeur) a été détecté en code review, pas en production.
- **Architecture simple et cohérente :** Aucune décision V1.0 n'a dû être remise en cause pour V1.1.

## Défis et leçons apprises

### Pièges d'outillage (non liés à la conception)

| Piège | Manifestation | Fix documenté |
|-------|--------------|---------------|
| `create-t3-app --CI` génère Pages Router | Conversion manuelle App Router requise | `feedback-pacal-setup.md` en mémoire |
| pnpm 11 bloque esbuild/sharp | Build échoue avec `ERR_PNPM_IGNORED_BUILDS` | `allowBuilds` dans `pnpm-workspace.yaml` |
| `NEXT_PUBLIC_*` sont build-time | Variables dans `environment` docker-compose sans effet | Retiré en code review |
| `<button>` dans `<Link>` | HTML invalide — détecté à l'implémentation | Restructuration carte en deux zones |
| `fs.existsSync` synchrone requis | `fs/promises` ne fonctionne pas dans le JSX react-pdf | Documenté dans la story 1.11 |

### Décision assumée

- **Pas de framework de test :** Décision d'architecture explicite, différée post-MVP. Non bloquante à ce stade pour un usage mono-utilisateur avec vérification TypeScript.
